# PR Automation And Fine-Tuning

ProPR turns implementation work into pull requests automatically, then keeps follow-up work in the pull request. You can review the result, leave normal GitHub comments, request AI review, apply AI review suggestions, switch models, update branches, and ask for conflict help without leaving the PR.

## Automatic Pull Request Creation

When ProPR finishes an implementation task, it handles the GitHub plumbing around it:

- Creates the feature branch (named with the issue number and model identifier)
- Commits the agent's changes
- Pushes to GitHub
- Opens a pull request linked to the source issue
- Posts status back to GitHub
- Updates task and label state (`<trigger>-processing` → `<trigger>-done`, or `<trigger>-failed-*` on failure)

This keeps the agent focused on code while ProPR handles the repeatable workflow around the code.

## Use ProPR On Any Pull Request

You do not have to let ProPR create a PR to use its review and fix tools. The pull request is an entry point on its own, so you can apply ProPR to PRs opened by a teammate, another agent, or yourself outside ProPR:

- **Review or fix only**: comment `/review` on any eligible PR to get AI review feedback, then `/fix` to apply it. Slash commands from an allowed author are processed directly — the PR does not need a processing label for commands to run.
- **Take over an existing PR**: add a configured processing label (for example `AI` or `propr`) to the open PR. From then on, normal follow-up comments are picked up just like on a ProPR-created PR, and ProPR continues the work in place.

The only difference between a ProPR-created PR and one you hand over is who opened it — once ProPR is engaged, follow-up comments and slash commands behave the same way.

## Natural Follow-Up Comments

For ordinary refinement, you do not need slash commands. Post a normal PR comment that describes the requested change:

```text
Please update the empty state copy and add a regression test for the loading spinner.
```

ProPR picks up the comment, includes PR context and the comment content, and queues the requested change for processing. How comments are picked up:

- The PR must carry a processing label (for example `AI` or `propr`), or the comment must contain a trigger keyword from `PR_FOLLOWUP_TRIGGER_KEYWORDS` (for example `!propr`).
- Bot comments are ignored; `GITHUB_USER_WHITELIST` and `GITHUB_USER_BLACKLIST` control who can trigger work.
- Comments posted while a job for the same PR is already running are batched and handled together once the active job finishes.
- Line-level review comments carry their file path, line, and diff hunk to the agent, so "fix this" on a specific line has real context.
- Images attached to comments are available to the agent.

When the follow-up task completes, ProPR posts a completion comment with a summary, the commit hash when changes were committed, and an expandable slash-command reference.

## Review And Fix Loop

Slash commands are reserved for specific PR actions:

- `/review` posts AI review feedback (severity findings plus a `Score: N/10` line) without changing code.
- `/fix` applies unprocessed AI review suggestions generated by `/review`.
- `/merge` merges the base branch into the PR branch and resolves conflicts with agent help.
- `/switch` changes the PR's configured model.
- `/use` runs one follow-up pass with a temporary model override.
- `/ultrafix` runs an automated review-fix loop.

The key distinction is direct user feedback versus AI review feedback. User-authored comments are processed immediately as natural follow-up instructions. `/fix` only deals with AI review comments generated by `/review`.

## Editable AI Review Feedback

AI review comments are intentionally human-editable before `/fix` runs:

- Delete suggestions you do not want applied.
- Rewrite suggestions that are directionally right but too vague.
- Leave only the findings worth implementing.

When `/fix` runs, ProPR collects the remaining unprocessed AI review comments and applies them in a single implementation pass, then marks them processed.

## Stronger Correction Loops

For more autonomous cleanup, `/ultrafix` alternates review and fix passes until the review score reaches the configured goal (default 7/10) or the maximum cycle count (default 5) is exhausted. It waits for CI checks and PR inactivity between cycles so it does not blindly race human updates.

The visible `ultrafix` PR label acts as a circuit breaker. Removing it stops the loop after the current cycle.

See [PR Slash Commands](./pr-commands.md) for full syntax and examples.
